# Time-Series Pipeline — Hourly Energy Consumption (Group 10)

An end-to-end pipeline for forecasting hourly electricity demand on the PJM grid,
built around the [Hourly Energy Consumption](https://www.kaggle.com/datasets/robikscube/hourly-energy-consumption)
dataset. We forecast **PJME** (PJM East) demand one hour ahead from its own recent
history and calendar features.

## Problem

PJME is a regional load series sampled every hour from 2002 to 2018 (~145k readings).
Grid operators need short-term demand forecasts to schedule generation, so the target
is the next hour's consumption in megawatts (`PJME_MW`). The series has strong daily,
weekly and yearly cycles, which makes it a good fit for lag and moving-average features.

## Tasks

- **Task 1 — Preprocessing, analysis & modeling** ✅ (`notebooks/`, `src/preprocessing.py`)
  - `01_eda.ipynb` — time range, granularity, missing values, distributions
  - `02_analysis.ipynb` — analytical questions (trend, seasonality, lag effects, moving averages, external correlation)
  - `03_modeling.ipynb` — model, hyperparameter tuning, experiment comparison
- **Task 2 — SQL + MongoDB design** —
I worked with the group on Task 2 of our group assignment where we were required to design and implement two databases out of the Hourly Energy Consumption dataset provided by Kaggle. I made a 3-table relational schema design for MySQL, then drew an ERD, wrote the SQL schema scripts and implemented 3 queries for the latest record, date range filtering, and average consumption by hour. I created a collection for MongoDB that had all the pertinent data associated with a single hourly reading (region, datetime, consumption value, and time breakdown fields), and I wrote those same 3 queries with the MongoDB syntax. Both implementations are structured, documented and suitable to be passed to the teammate working on Task 3, who will develop the API on top of them.

- **Task 3 — CRUD API** ✅ (`task3/`)
I built the Task 3 API on top of the SQL and MongoDB designs from Task 2, using FastAPI. The app lives in `task3/`, with `sql_api/` and `mongo_api/` as parallel packages (`database.py` for the connection, `models.py` for the Pydantic request/response schemas, `routes.py` for the endpoints), plus a shared `utils.py` that derives the same calendar features (year, month, hour, day_of_week, is_weekend, season) used when Task 2's data was bulk-loaded, so records created through the API line up with the existing rows/documents. Each database exposes the same resource shape at `/api/sql/readings` and `/api/mongo/readings`: full CRUD (`POST`, `GET` by id, `PUT`, `DELETE`) plus the two required time-series endpoints, `GET /latest` and `GET /range?start=...&end=...`. Interactive docs are auto-generated by FastAPI at `/docs`. I tested every SQL endpoint end-to-end against a real local MySQL instance (create, read, update, delete, latest, and range all verified); the MongoDB endpoints are built on the identical pattern and use the same route/schema code, but I wasn't able to fully verify them live before this handoff due to local environment issues getting MongoDB running — worth a final check before submission.
- **Task 4 — Forecast script** — _in progress_ (imports `src/preprocessing.py`)

### Running the Task 3 API

```bash
cd task3
pip install -r requirements.txt
# point these at your own MySQL / MongoDB instances if different from the defaults
set MYSQL_HOST=localhost
set MYSQL_PASSWORD=your_password
set MONGO_URI=mongodb://localhost:27017/
uvicorn main:app --reload
# docs at http://127.0.0.1:8000/docs
```
